How It Works

Execution Flow

Phase 1 — Decompose
Opus (coordinator) reads the codebase, understands the request, and breaks it into workstreams. Each workstream gets: a clear scope, file boundaries, and a model assignment based on strengths.

Phase 2 — Build (parallel)
Each model gets its own git worktree and works independently. They can only touch files in their assigned scope. Real-time progress streams back to the swarm UI.

Phase 3 — Wire + Merge
The coordinator merges all worktrees, resolves conflicts, adds integration code (shared types, API clients, error handling), and runs tests.

Phase 4 — Review
Optional: run a deliberation round where all models review the merged result and flag issues.

New Providers

OpenAI Direct — API key stored in macOS Keychain (Void-openai). Calls OpenAI API directly (not via OpenRouter). Needed for Codex integration — same auth flow the codex plugin uses, but native.

Google Gemini — API key stored in Keychain (Void-gemini). Calls generativelanguage.googleapis.com directly. Used by the Designer Agent and available as a model in the picker.

Both also fall back to env vars: OPENAI_API_KEY, GEMINI_API_KEY.

Relationship: Deliberation vs Swarm

Deliberation → Think

/deliberate "how should we..."

Models debate sequentially. Output is a plan or recommendation. No code written. You decide what to implement.

Input: question → Output: answer

Swarm → Build

/swarm "build a dashboard with..."

Models work in parallel on isolated worktrees. Output is actual code, merged and tested. Coordinator handles integration.

Input: spec → Output: working code

They compose: /deliberate first to design, then /swarm to build the winning approach.
