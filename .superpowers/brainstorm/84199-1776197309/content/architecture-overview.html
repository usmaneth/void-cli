Void CLI — Feature Architecture

Four interconnected systems. Two approaches to wire them together.

A

Plugin-First — Everything as Plugins

All four features ship as separate plugins in a Void marketplace repo you own. OpenRouter browser, Deliberation Room, Gemini Designer, and Provider Config each live in their own plugin with agents, skills, and scripts.

Pros

  • Zero changes to Void core — stays in sync with upstream Claude Code
  • Each feature is independently installable/removable
  • Other Void users could install your plugins
  • Fast to ship — plugin structure is well-defined

Cons

  • Model picker UI can't be modified from a plugin — it's hardcoded in core
  • Deliberation room needs deep integration with the main loop
  • Some features will feel bolted-on rather than native
  • Limited to Bash + skills — no React component access
B

Core-Native — Build Into Void's Source

All four features are built directly into Void's codebase. Model picker gets a real OpenRouter browser component. Deliberation room is a new execution mode alongside council. Designer agent is a built-in agent type. Provider config gets a proper settings UI.

Pros

  • Fully native UX — model picker, hotkeys, settings all feel integrated
  • Direct access to React components (Ink), state management, streaming
  • Deliberation room can hook into the main conversation loop
  • Single build, single binary — no plugin dependency chain

Cons

  • Harder to merge upstream Claude Code updates (more divergence)
  • Larger codebase to maintain
  • All-or-nothing — can't disable individual features easily
C

Hybrid — Core for UI, Plugins for Agents ★ Recommended

Split by what needs core access vs what works great as a plugin. Model picker + deliberation mode + provider config go into Void's source (they need React/Ink and main loop access). Designer agent ships as a plugin (it's self-contained — just an agent definition + Gemini SDK script).

Pros

  • Native UX where it matters (model picker, deliberation, hotkeys)
  • Designer agent is portable — works with any Claude Code fork
  • Clean separation: core handles orchestration, plugin handles design AI
  • Manageable upstream divergence — only UI/model code diverges

Cons

  • Two places to maintain code (core + plugin)
  • Need to coordinate versioning between core and plugin
